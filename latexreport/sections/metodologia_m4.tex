\subsection{Milestone 4 -- Refactoring automatizzato}

La Milestone 4 caratterizza l'efficacia del refactoring automatizzato assistito da LLM
su classi Java di Apache OpenJPA. Il processo si articola in due fasi: selezione delle
classi target (Step~8) e produzione iterativa di versioni refactored con vincoli di test
progressivamente più stringenti (Step~9).

\subsubsection{Selezione delle classi (Step 8)}

L'analisi si concentra sull'ultima release disponibile nel repository Git di Apache
OpenJPA (\texttt{4.1.1}).

\paragraph{Analisi PMD}

Tutte le classi Java di produzione della release vengono analizzate con PMD 7.7.0
tramite gli stessi quattro ruleset adottati in Milestone 1
(\texttt{bestpractices}, \texttt{design}, \texttt{errorprone}, \texttt{codestyle}).
Per ogni classe viene contato il numero totale di violazioni (\textbf{Nsmells}),
e per le due classi selezionate viene prodotto un report dettagliato di ogni violazione
(nome regola, categoria, numero di riga, messaggio) tramite \texttt{SmellDetailService}.

\paragraph{Filtri applicati prima della selezione}

Le classi sono escluse se soddisfano uno dei tre criteri seguenti, nell'ordine:

\begin{enumerate}
  \item \textbf{Dimensione}: LOC $< 100$ -- classi troppo piccole con poche opportunità
        di refactoring.
  \item \textbf{Tipo non istanziabile}: interfacce (\texttt{interface},
        \texttt{sealed interface}, \texttt{non-sealed interface}) e classi astratte
        (\texttt{abstract class}) -- prive di implementazione completa, non candidati
        significativi per un LLM che deve rimuovere smell strutturali.
  \item \textbf{Nsmells minimi}: Nsmells $< 10$ -- classi sostanzialmente pulite che
        non offrono materiale sufficiente per un refactoring motivato. Questa soglia
        garantisce che la classe selezionata come ``last\,-2'' abbia abbastanza smell da
        rendere il prompt Copilot non banale.
\end{enumerate}

\paragraph{Ordinamento e formula di selezione}

Le classi rimanenti sono ordinate per Nsmells decrescente; a parità di Nsmells il
criterio secondario è LOC decrescente. La selezione avviene tramite la formula del
docente:

\begin{enumerate}
  \item Prima lettera del nome: \texttt{G} (Giordano) $\Rightarrow$ numero = 7
  \item $X = 7 \bmod 5 = 2$
  \item Caso 2: si selezionano le classi a rank $1{+}X = 3$ e $N{-}X = N{-}2$
\end{enumerate}

dove $N$ è la dimensione dell'elenco dopo tutti i filtri.

\paragraph{Output della selezione}

La pipeline produce due file in \texttt{output/milestone4/}:
\begin{itemize}
  \item \texttt{class\_selection.csv} -- lista completa delle $N$ classi filtrate con
        rank, path, Nsmells, LOC e flag di selezione;
  \item \texttt{smell\_report.txt} -- per ciascuna delle due classi selezionate: tabella
        di ogni violazione PMD (regola, categoria, righe, messaggio) e uno scheletro
        del prompt Copilot pronto all'uso.
\end{itemize}

\subsubsection{Refactoring automatizzato (Step 9)}

Per ogni classe selezionata viene prodotta una sequenza di versioni refactored
($C_1$, $C_2$, $C_3$, $C_4$) tramite Microsoft Copilot (account universitario).
Il prompt specifica di:

\begin{enumerate}
  \item rimuovere i code smell identificati da SonarCloud senza alterare la funzionalità;
  \item superare i test forniti;
  \item non introdurre modifiche diverse da quelle richieste;
  \item garantire la compatibilità con il resto del sistema.
\end{enumerate}

Ogni versione aggiunge un livello di test al prompt, come mostrato in
Tabella~\ref{tab:m4-versions}.
